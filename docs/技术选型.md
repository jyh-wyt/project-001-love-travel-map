## 4. AI 智能应用开发岗方向

用户进一步说明：以后更倾向于 AI 智能应用开发岗。

因此本项目后续应逐步从“传统全栈项目”升级为“AI + 全栈项目”。但第一版不应一开始就堆 AI 功能，而是先把用户、旅行记录、图片、计划、权限、部署这些基础能力做好，再在第二阶段加入 AI 能力。

推荐长期技术组合：

1. 前端：Next.js，负责智能交互界面。
2. 后端：Spring Boot，负责业务接口、权限、数据和 AI 服务编排。
3. 数据库：MySQL，保存用户、空间、旅行、计划、记录等结构化数据。
4. 缓存：Redis，保存验证码、会话、限流、短期任务状态。
5. 对象存储：阿里云 OSS，保存旅行照片。
6. AI 能力：后续接入大模型 API，用于旅行日记润色、旅行总结、计划建议、图片描述、智能搜索等。
7. 向量检索：后续可加入向量数据库或向量检索能力，用于“按语义搜索旅行记忆”。

AI 功能建议分阶段加入。用户已确认：第一版完成后，可以先做 AI 增强，再补企业工程化能力。

| 阶段 | AI 能力 | 说明 |
| --- | --- | --- |
| 第一阶段 | 日记润色 | 用户写完旅行日记后，AI 帮助润色成更自然的文字 |
| 第二阶段 | 旅行总结 | 根据一段旅行的多条记录生成总结 |
| 第二阶段 | 计划建议 | 根据目的地、日期和偏好生成行程建议 |
| 第二阶段 | 图片描述 | 为上传照片自动生成描述或标签 |
| 第二阶段 | 智能搜索 | 用户输入“那次下雨去海边的旅行”，系统能找出相关记录 |
| 第三阶段 | Docker / Nginx / 监控等 | 补齐企业工程化与部署能力 |

第一版已确认：

```text
第一版预留最小 AI 入口。
```

推荐实现方式：

1. 前端在日记编辑页预留“AI 润色”按钮。
2. Java 后端保留 AI 调用接口，例如 `/api/ai/polish-diary`。
3. Python 单独写 AI 服务，负责调用大模型 API 或执行 AI 逻辑。
4. 第一版直接接入真实大模型 API，不使用模拟润色结果。

推荐调用链：

```text
前端 React/Next.js
  -> Java Spring Boot 业务后端
  -> Python AI 服务
  -> 大模型 API
```

这样可以同时展示：

1. Java 后端业务开发能力。
2. Python AI 应用开发能力。
3. 前后端分离能力。
4. 多服务协作能力。
5. 后续接入大模型、向量检索和智能搜索的扩展能力。

已确认 AI 调用架构采用方案 B：

```text
前端 Next.js
  -> Java Spring Boot 统一业务后端
  -> Python FastAPI AI 服务
  -> LangChain / 大模型 API
```

含义：

1. 前端只直接调用 Java Spring Boot。
2. Java 后端负责登录、权限、情侣空间校验、AI 调用次数限制、业务数据保存。
3. Python FastAPI 只负责 AI 能力，例如日记润色、旅行总结、计划建议。
4. Python AI 服务不直接暴露给前端。
5. Java 后端通过 HTTP 调用 Python AI 服务。

第一版 AI 日记润色已确认直接接真实大模型 API，不使用模拟结果。API Key 必须放在 Python AI 服务或后端部署环境变量中，不能暴露到前端。

### 7.5 地图实现方式

已确认第一版：

```text
Apache ECharts 地图可视化
```

第一版实现目标：

1. 首页展示可缩放、可拖动的中国地图。
2. 初始层级主要展示省份。
3. 用户放大或点击省份后，展示该省的城市记录标记。
4. 去过的城市使用高对比度、色盲友好的标记，并可显示旅行代表照片。
5. 点击城市标记后进入对应地区的旅行记录页面。
6. 第一版不接入百度地图或高德地图，因此不需要地图 AK，也没有在线地图调用额度成本。

能力边界：

1. ECharts 适合展示行政区域、去过地区、照片标记和数据交互。
2. 第一版不提供道路、街道、实时定位和路线导航。
3. 后续需要真实地理底图或导航时，可以在保留旅行记录数据结构的基础上替换为百度地图或高德地图。
