# MVP 开发边界

## 1. 第一版目标

第一版目标是跑通完整产品链路，而不是一次性做满全部功能。

第一版需要同时覆盖：

1. 前端页面。
2. Java 业务后端。
3. MySQL 数据保存。
4. Redis 基础使用。
5. 阿里云 OSS 图片上传。
6. Python FastAPI AI 服务。
7. LangChain 调用阿里云百炼。
8. 前端通过 Java 后端间接调用 AI 润色。

## 2. 第一版功能范围

第一版做：

1. 情侣空间。
2. 匿名体验。
3. 邀请码加入。
4. 中国地图首页。
5. 去过地区标记。
6. 地区旅行记录流。
7. 图文日记发布。
8. 图片上传到 OSS。
9. 日历计划。
10. AI 日记润色。

情侣空间的作用：

1. 不是单独页面概念，而是共享数据容器。
2. 两个人绑定后，共享地图地区、旅行记录、图片、日记和计划。
3. 所有共享数据必须归属于同一个 `space_id`。

内容编辑规则：

1. 新增、编辑、发布都需要点击 `确定` 或 `发布` 后才保存。
2. 点击 `取消` 则放弃本次修改。
3. 第一版不做边输入边保存。
4. 旅行记录只有一个发布入口，图片和日记一起填写、一起发布。
5. 第一版一条旅行记录最多 6 张图片。
6. 点击地图地区后进入地区记录新页面。
7. 地图体验参考百度地图的缩放和拖动手感，但层级切换采用点击省份下钻。
8. 第一版地图技术采用 Apache ECharts，不接入百度地图或高德地图。
9. 初始展示全国省份地图，点击省份后进入该省城市地图，再点击城市进入地区记录页。
10. 城市层级通过省级下钻和城市标记实现，不包含道路、街道、定位和导航能力。
11. 地区旅行记录页采用一个城市一个记录流。
12. 旅行记录页上方提供旅行日期范围选择，保存为 `start_date` 和 `end_date`。
13. 第一版支持手动将某张旅行图片设为城市代表照片。
14. 如果用户还没有手动设置封面，可先使用第一条记录的第一张图片作为默认展示。

## 3. 第一版登录方式

已确认：

```text
匿名体验 + 邀请码
```

含义：

1. 用户第一版不需要注册账号。
2. 创建情侣空间时生成一个匿名用户身份。
3. 系统生成邀请码。
4. 另一半输入邀请码后加入同一个情侣空间。
5. 两个人可以共同查看和编辑旅行记录、照片和计划。

底层账号设计：

1. 使用 `app_user` 用户主表。
2. 使用 `user_auth_identity` 保存登录身份。
3. 第一版匿名身份作为一种 `identity_type = anonymous` 的登录身份。
4. 后续可升级邮箱、手机号、微信登录，不推翻已有业务表。

已确认细节：

1. 匿名身份保存在浏览器 localStorage。
2. 情侣空间第一版限制 2 人。
3. 邀请码有效期为 24 小时。
4. 第一版图片不做缩略图。
5. AI 润色日志保存完整输入和输出。
6. AI 日记润色限制为每人每天 10 次。
7. OSS 上传方式为前端拿签名后直传 OSS。
8. 删除旅行时，旅行、日记、图片元数据都做逻辑删除。
9. 删除图片时，同步删除 OSS 文件，避免后期存储堆积。
10. 绑定入口为 C：首次进入弹出绑定引导，底部“我们”也可进入。
11. 对方加入成功后，邀请码立刻失效。

## 4. 第一版暂不做

1. 邮箱注册登录。
2. 手机号登录。
3. 微信登录。
4. 找回密码。
5. 公开社区。
6. 评论点赞。
7. 地图导航。
8. 复杂权限体系。
9. 多情侣空间。
10. 复杂 AI Agent。
11. RAG。
12. 向量数据库。
13. Docker / Kubernetes。
14. 完整监控系统。
15. 百度地图或高德地图在线底图。
16. 道路、街道、实时定位和路线导航。

## 5. 后续升级路径

第一版完成后，可以升级：

1. 匿名用户绑定正式账号。
2. 邮箱或手机号登录。
3. 更完整的权限体系。
4. AI 旅行总结。
5. AI 行程建议。
6. 语义搜索旅行记忆。
