# 免费部署资源筛选清单

更新时间：2026-06-24

说明：免费额度和限制经常变化，真正部署前必须再次查看官网。本文只用于当前项目测试部署选型，不作为长期生产承诺。

## 1. 当前结论

当前项目不是一个纯静态前端，它包含：

1. Next.js 前端。
2. Java Spring Boot 后端。
3. Python FastAPI AI 服务。
4. MySQL。
5. Redis。
6. OSS。

所以“全免费部署”比较难，尤其是 Java、Python、MySQL、Redis 同时长期运行时。当前更现实的免费优先策略是：

```text
前端尽量免费
后端和数据库先找免费试用或低成本服务器
AI 调用严格限流
图片继续使用 OSS 并控制上传量
```

## 2. 前端免费平台

### 2.1 Cloudflare Pages

适合程度：高。

适合放：

1. 前端静态资源。
2. 静态导出的页面。
3. 测试域名和预览部署。

官方免费限制中可关注：

1. 免费计划每月构建次数有限制。
2. 单个资源文件大小有限制。
3. 免费计划支持一定数量的自定义域名。
4. 预览部署数量较宽松。

对本项目的判断：

1. 如果前端可以作为静态页面或只请求 Java API，Cloudflare Pages 很适合。
2. 如果 Next.js 用到复杂服务端能力，需要确认 Cloudflare Pages 对该模式的支持。
3. 国内访问速度和稳定性需要实际测试。

参考：

```text
https://developers.cloudflare.com/pages/platform/limits/
```

### 2.2 Vercel

适合程度：中到高。

适合放：

1. Next.js 前端。
2. 预览部署。
3. GitHub 自动部署。

官方免费限制中可关注：

1. Hobby 计划有项目数、每日部署数、构建时长、流量等限制。
2. Next.js 支持很好。
3. 超出免费额度或使用部分高级能力可能收费。

对本项目的判断：

1. 因为本项目前端是 Next.js，Vercel 体验会很好。
2. 但后端 Java、Python、MySQL、Redis 不适合全部塞进 Vercel。
3. 如果只把前端放 Vercel，后端放服务器，是可行方案。
4. 国内访问速度需要实际测试。

参考：

```text
https://vercel.com/docs/limits
```

### 2.3 Netlify

适合程度：中。

适合放：

1. 静态前端。
2. 简单前端预览。

对本项目的判断：

1. 如果只部署静态前端，可以考虑。
2. 对 Next.js 的完整服务端能力和国内访问情况，需要实际测试。
3. 本次没有把它作为第一推荐。

## 3. 后端免费平台

### 3.1 Render Free

适合程度：中。

适合放：

1. 测试 Web Service。
2. Python FastAPI 小服务。
3. 简单后端预览。
4. 免费 Postgres 或 Key Value 体验。

官方说明中可关注：

1. Render 提供免费 Web Service、Postgres、Key Value、静态站点等类型。
2. 免费 Web Service 闲置一段时间会休眠，再次访问需要冷启动。
3. 免费服务的本地文件系统是临时的，重启或重新部署后本地文件会丢失。
4. 免费数据库可能有有效期限制。
5. 免费实例不建议用于生产。

对本项目的判断：

1. Python FastAPI AI 服务可以尝试放 Render 免费服务测试。
2. Java Spring Boot 也可以研究是否能用 Docker 或 Web Service 跑，但首次部署可能不如云服务器直观。
3. MySQL 和 Redis 与当前项目技术栈不完全匹配，Render 免费数据库主要不是 MySQL。
4. 休眠会影响体验，第一次访问可能慢。

参考：

```text
https://render.com/docs/free
```

## 4. 数据库与 Redis 免费选择

### 4.1 MySQL

免费优先选择：

1. 测试阶段在一台免费试用服务器或低成本服务器上自建 MySQL。
2. 如果云厂商提供 RDS 免费试用，可以短期体验。

不建议：

1. 把正式数据放在随时可能过期的免费数据库里。
2. 没有备份就长期使用免费数据库。

### 4.2 Redis

免费优先选择：

1. 测试阶段在服务器自建 Redis。
2. 不开放公网端口。
3. 只允许后端本机或内网访问。

不建议：

1. 免费 Redis 暴露公网。
2. 用没有密码、没有防火墙的 Redis。
3. 把 Redis 当成长期数据存储。

## 5. OSS 和 AI 成本控制

### 5.1 OSS

当前项目继续使用阿里云 OSS。

控制成本方式：

1. 限制单张图片大小。
2. 删除记录时同步删除 OSS 对象。
3. 测试阶段少量上传图片。
4. 后续再考虑缩略图和 CDN。

### 5.2 AI

当前项目使用通义千问。

控制成本方式：

1. 普通用户 30 天 3 次 AI 规划。
2. VIP 用户 30 天 10 次 AI 规划。
3. Redis 防重复点击。
4. Redis 短窗口限流。
5. 测试时不要反复生成长内容。

## 6. 当前推荐组合

### 6.1 最省钱测试组合

```text
前端：Cloudflare Pages 或 Vercel 免费层
Java 后端：本地或免费试用服务器
Python AI 服务：同一台服务器，或 Render 免费服务测试
MySQL：服务器自建
Redis：服务器自建
OSS：已有阿里云 OSS，少量测试
域名：先用平台二级域名或服务器 IP
HTTPS：前端平台自动 HTTPS；后端正式域名后再配置
```

优点：

1. 成本最低。
2. 能学习完整部署链路。
3. 前端能先拿到一个公开访问地址。

缺点：

1. 后端如果没有免费服务器，仍可能需要低成本服务器。
2. 多平台组合会有跨域和环境变量配置。
3. Render 免费服务可能休眠。

### 6.2 最容易理解的测试组合

```text
一台云服务器：
  Nginx
  Next.js
  Spring Boot
  FastAPI
  MySQL
  Redis
OSS 使用阿里云
```

优点：

1. 架构最直观。
2. 适合第一次学习 Linux、Nginx、端口和环境变量。
3. 不容易被多个免费平台的限制绕晕。

缺点：

1. 如果没有免费试用服务器，需要少量服务器费用。
2. 所有服务在一台机器，生产稳定性一般。

## 7. 筛选检查表

选择免费资源前逐项确认：

```text
[ ] 是否支持 Java 17
[ ] 是否支持长期运行 Spring Boot
[ ] 是否支持 Python FastAPI
[ ] 是否支持 MySQL，还是只能用 Postgres
[ ] 是否支持 Redis，还是只能用别的 Key Value
[ ] 免费服务是否会休眠
[ ] 免费数据库是否会过期
[ ] 是否支持环境变量
[ ] 是否支持自定义域名
[ ] 是否支持 HTTPS
[ ] 是否限制构建次数
[ ] 是否限制流量
[ ] 国内访问速度是否能接受
[ ] 超出免费额度是否自动扣费
[ ] 是否需要绑定信用卡
```

## 8. 当前建议

先做两步：

1. 前端优先尝试 Cloudflare Pages 或 Vercel 免费部署。
2. 后端、MySQL、Redis、Python AI 服务先找免费试用服务器；如果没有合适免费资源，再考虑低成本单服务器测试部署。

不建议一开始就把所有服务拆到多个云产品里，因为：

1. 配置复杂。
2. 成本更难控制。
3. 新手第一次部署容易卡在跨域、内网、端口、安全组、环境变量。

## 9. 面试表达

可以这样说：

> 我在部署方案上做了免费优先和生产架构的区分。测试阶段优先使用 Cloudflare Pages 或 Vercel 免费部署前端，后端、MySQL、Redis、Python AI 服务先用免费试用服务器或低成本服务器跑通。正式阶段再迁移到 RDS、云 Redis、独立域名、HTTPS 和自动化部署。这样既控制学习成本，也保留了向企业级架构演进的路径。
