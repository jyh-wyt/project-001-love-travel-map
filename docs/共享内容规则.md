# 共享内容规则

## 1. 情侣空间的含义

`情侣空间` 不是一个单独的展示功能，而是所有共享内容的归属容器。

两个人通过邀请码绑定到同一个情侣空间后，共享以下内容：

1. 地图上去过的地区。
2. 每个地区的旅行记录。
3. 旅行照片。
4. 旅行日记。
5. 日历旅行计划。
6. 后续 AI 润色结果。

所有共享数据都必须关联同一个：

```text
space_id
```

## 1.1 绑定入口规则

已确认：

```text
绑定入口选择 C：新用户首次打开时弹出绑定引导，同时底部导航“我们”也可以随时进入绑定/空间页面。
```

未绑定状态：

1. 首页可以显示绑定引导。
2. 底部导航“我们”可以进入绑定页面。
3. 用户可以选择创建情侣空间，或输入对方邀请码加入空间。

已绑定状态：

1. “我们”页面展示情侣空间信息。
2. 地图、记录、计划都展示该情侣空间下的共享内容。

## 1.2 邀请码使用规则

已确认：

```text
对方加入成功后，邀请码立刻失效，状态改为 used。
```

规则：

1. 邀请码为 6 位大写字母数字。
2. 有效期 24 小时。
3. 情侣空间最多 2 人。
4. 加入成功后邀请码立刻不可再次使用。

## 2. 地图共享规则

用户在地图上点击某个地区，例如：

```text
青岛
```

系统应弹出或进入青岛的旅行记录页面。

青岛旅行记录中包含：

1. 图片。
2. 日记。
3. 旅行时间。
4. 关联计划。

如果用户 A 在青岛发布图片或日记，用户 B 进入同一个情侣空间后也能看到。

如果用户 B 在青岛发布图片或日记，用户 A 也能看到。

已确认地图交互方向：

```text
地图需要像百度地图一样可以缩放。
```

缩放规则：

1. 初始缩放级别主要显示省份。
2. 继续缩放后显示城市。
3. 用户可以点击城市，例如青岛，进入该城市的旅行记录。
4. 第一版支持城市和省份两级概念，但具体地图技术方案仍需确认。

已确认地区记录打开方式：

```text
点击地区后进入新页面。
```

## 3. 计划共享规则

旅行计划属于情侣空间。

用户 A 填写计划后，用户 B 可以同步看到。

用户 B 修改计划后，用户 A 也可以看到。

计划内容包括：

1. 日期。
2. 时间安排。
3. 每天去哪。
4. 备注。

## 4. 确认 / 取消规则

第一版不做边输入边保存。

所有新增、编辑、发布类操作都需要明确按钮：

```text
确定
取消
```

规则：

1. 点击 `确定` 后，才向后端提交并保存。
2. 点击 `取消` 后，放弃本次输入或修改。
3. 用户输入过程中不自动保存到数据库。
4. 发布图文日记前需要点击 `发布` 或 `确定`。
5. 编辑旅行计划前需要点击 `确定` 才保存。

已确认旅行记录发布方式：

```text
记录页面只有一个发布入口。图片和日记一起填写，一起发布。
```

规则：

1. 用户点击一个“发布记录”按钮。
2. 进入或打开统一编辑界面。
3. 在同一个界面填写日记文字。
4. 在同一个界面选择图片。
5. 第一版一条记录最多 3 张图片。
6. 点击 `发布` / `确定` 后，图片和文字作为同一条记录保存。
7. 点击 `取消` 后，文字和图片都不保存。

## 5. 后端数据原则

后端所有共享内容查询必须按 `space_id` 过滤。

不能只根据地区名称查询，必须同时带上情侣空间：

```text
space_id + region
```

示例：

```text
查询青岛记录 = 当前情侣空间 space_id + 青岛 region
```

这样不同情侣之间的数据不会互相看到。
