# 部署前检查清单

## 1. 文档用途

以后每次上线前都按这份文档检查，避免漏配环境变量、数据库、Redis、OSS、AI Key、跨域和端口。

这份文档也可以用于面试说明：

> 我在项目中沉淀了部署前检查清单，把配置、构建、数据库迁移、缓存、对象存储、AI 服务、跨域、安全项和上线后验证都标准化，避免靠记忆上线。

## 2. 上线前总流程

```text
确认代码版本
  -> 检查环境变量
  -> 检查数据库和迁移
  -> 检查 Redis
  -> 检查 OSS
  -> 检查 DashScope
  -> 构建前端
  -> 构建 Java 后端
  -> 检查 Python AI 服务
  -> 配置 Nginx 或部署平台
  -> 上线后冒烟测试
  -> 记录问题和修复
```

## 3. 代码与分支检查

上线前必须确认：

```text
[ ] 当前代码是准备上线的版本
[ ] 没有临时调试代码
[ ] 没有把真实密码、AccessKey、API Key 写进代码
[ ] 没有把 `.env`、本地日志、target、node_modules 提交
[ ] 前端、Java 后端、Python AI 服务都能在本地跑通
```

## 5. 数据库检查

```text
[ ] 线上 MySQL 已创建 `love_travel` 数据库
[ ] 线上 MySQL 账号不是 root 或已确认只是测试环境
[ ] 线上 MySQL 密码是强密码
[ ] 所有迁移 SQL 已执行
[ ] 用户表、空间表、旅行记录表、计划表、AI 额度表存在
[ ] 本地测试数据不会误导入正式环境
[ ] 数据库字符集使用 utf8mb4
```

重点验证：

```text
[ ] 注册用户成功
[ ] 用户名和账号重复校验正常
[ ] 登录成功后 Session 正常写入 Redis
[ ] 切换空间后地图和计划按当前空间隔离
```

## 6. Redis 检查

```text
[ ] Redis 服务已启动
[ ] Java 后端能连接 Redis
[ ] Redis 线上不暴露公网
[ ] Redis 密码已配置
[ ] 登录后能写入 Session
[ ] AI 规划额度、防重复点击、限流能写入 Redis
```

需要避免的问题：

1. 缓存穿透：无效请求要有限流和参数校验。
2. 缓存击穿：热点 Key 后续可用互斥锁或短暂锁保护。
3. 缓存雪崩：后续如果大量缓存 Key 设置过期时间，应增加随机过期时间。

## 7. OSS 检查

```text
[ ] Bucket 已创建
[ ] Endpoint 和 Bucket 地域一致
[ ] Java 后端配置了 AccessKey
[ ] AccessKey 没有写进前端
[ ] 上传小图片成功
[ ] 发布日记后图片能显示
[ ] 删除图片后 OSS 对象同步删除
[ ] 删除日记或城市记录后关联 OSS 图片处理符合预期
```

安全建议：

```text
[ ] 正式上线前轮换已经暴露过的 AccessKey
[ ] 使用 RAM 子账号
[ ] 子账号只授予当前 bucket 必要权限
```

## 8. DashScope 和 AI 服务检查

```text
[ ] Python AI 服务能启动
[ ] Python AI 服务监听 127.0.0.1:8000 或服务器内网地址
[ ] Java 后端 `AI_SERVICE_BASE_URL` 指向 Python AI 服务
[ ] DashScope API Key 已配置
[ ] 模型名为 `qwen-plus`
[ ] AI 规划接口能流式返回
[ ] 普通用户 30 天 3 次限制正常
[ ] VIP 用户 30 天 10 次限制正常
[ ] 重复点击不会重复消耗多次额度
```

注意：

1. Python AI 服务不建议直接暴露公网。
2. 前端只访问 Java 后端。
3. Java 后端负责权限、空间、额度、限流，再转发到 Python。

## 9. 前端构建检查

在 `apps/front` 执行：

```powershell
npm.cmd run build
```

检查：

```text
[ ] Next.js 构建通过
[ ] 没有 TypeScript 错误
[ ] `NEXT_PUBLIC_API_BASE_URL` 是线上后端地址
[ ] 登录页、地图页、计划页、空间页、设置页都能访问
[ ] 手机端主要页面不横向溢出
```

## 10. Java 后端构建检查

在 `apps/server` 执行：

```powershell
mvn clean package -DskipTests
```

检查：

```text
[ ] Maven 打包通过
[ ] jar 能启动
[ ] 端口为 `SERVER_PORT`
[ ] 数据库连接成功
[ ] Redis 连接成功
[ ] OSS 状态接口正常
[ ] AI 服务地址可访问
```

## 11. Python AI 服务检查

在 `apps/ai-service` 执行：

```powershell
python -m compileall app
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

检查：

```text
[ ] Python 编译通过
[ ] FastAPI 能启动
[ ] DashScope Key 未配置时能返回明确错误
[ ] DashScope Key 配置后 AI 规划能生成
```

## 12. 跨域和 Cookie 检查

因为项目使用 HttpOnly Cookie 登录 Session，上线时要特别检查：

```text
[ ] 前端域名已经加入 `FRONTEND_ALLOWED_ORIGINS`
[ ] 前端请求携带 credentials
[ ] 后端 CORS 允许 credentials
[ ] 登录响应能写入 Cookie
[ ] 刷新页面后仍保持登录
[ ] 未登录访问私密接口会返回 401 并跳转登录页
```

如果前后端不是同一个域名，Cookie 的 SameSite、Secure、HTTPS 还需要在正式上线前单独检查。

## 13. Nginx 或部署平台检查

如果使用阿里云轻量服务器：

```text
[ ] 80/443 端口开放
[ ] Java 后端端口不直接暴露给公网，或只在测试阶段临时开放
[ ] Python 8000 不开放公网
[ ] Nginx 正确转发前端和 `/api`
[ ] 后续配置 HTTPS 证书
```

如果只部署前端到 Netlify / Cloudflare Pages：

```text
[ ] 构建目录和构建命令正确
[ ] `NEXT_PUBLIC_API_BASE_URL` 已配置
[ ] 后端如果还在本地，公网用户无法使用完整业务链路
[ ] 这只算免费试水，不算完整上线
```


## 15. 部署方式选择

当前推荐分两步走。

第一步：免费试水。

```text
只部署前端到 Netlify 或 Cloudflare Pages
目标是熟悉构建、环境变量、线上访问流程
不要求完整业务可用
```

第二步：真正可用。

```text
使用阿里云轻量服务器部署全套
Nginx + Next.js + Spring Boot + FastAPI + MySQL + Redis + OSS + DashScope
目标是形成可演示、可面试讲解的完整项目
```

## 16. 面试讲法

可以这样讲：

> 这个项目不是只在本地跑起来。我把部署配置拆成前端、Java 后端、Python AI 服务、MySQL、Redis、OSS、DashScope 七类，并写了部署前检查清单。上线前会检查环境变量、数据库迁移、Redis Session、OSS 上传删除、AI 流式接口、跨域 Cookie、构建结果和核心业务冒烟测试。这样能降低配置遗漏，也体现了我对全栈项目交付流程的理解。
