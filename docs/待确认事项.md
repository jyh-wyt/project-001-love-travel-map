## 2. 技术选型

已确认：

1. 第一版选择路线 B：稍微接近真实产品，一开始就接数据库和图片存储。
2. 用户已有 MySQL、Redis、JavaWeb、Spring Boot、MyBatis-Plus、阿里云 OSS 使用经验。
3. 用户希望以后就业开发岗，因此项目技术路线应尽量贴近企业常见后端开发实践。
4. 当前倾向使用 Spring Boot + MySQL + Redis + MyBatis-Plus + 阿里云 OSS。
5. 用户以后更倾向于 AI 智能应用开发岗，因此项目后续应逐步升级为 AI + 全栈项目。
6. 第一版预留最小 AI 入口。
7. AI 方面接口可以用 Python 写，并在 PyCharm 运行。
8. 传统后端部分用 Java / Spring Boot 写，并在 IntelliJ IDEA 运行。
9. 前端继续在 VS Code 运行。
10. Nginx 当前不熟，需要后续从基础配置讲起。
11. Python AI 服务框架选择 FastAPI。
12. 用户当前正在学习 LangChain，后续 AI 功能可以考虑用 LangChain 编排。
13. 前端框架选择 Next.js。
14. AI 调用架构采用方案 B：前端只调用 Java，Java 再调用 Python FastAPI AI 服务。
15. 第一版 AI 日记润色直接接真实大模型 API，不使用模拟结果。
16. 第一版大模型服务商选择阿里云百炼 / 通义千问。
17. Python AI 服务通过 LangChain 调用阿里云百炼 / 通义千问。
18. 第一版模型名称暂定 `qwen3.5-omni-plus`。
19. 模型名称需要做成可配置项，方便用户后续自己更换。
20. 第一版完成后，优先做 AI 增强，再补企业工程化能力。
21. 第一版地图实现正式选择 Apache ECharts。
22. 第一版不接入百度地图或高德地图，不承担地图 AK 和在线调用额度成本。

待确认：

1. 后端路线是否正式定为 Spring Boot + MySQL + Redis + MyBatis-Plus。
2. 图片存储是否正式定为阿里云 OSS。
3. 是否第一版就做 PWA。
4. 单次 AI 日记润色的输入长度和输出长度限制。
5. API Key 和模型配置文件的具体命名方式。
6. 第一版完成后，第二阶段 AI 增强优先做哪个能力。

## 3. 页面结构

已确认方向：

1. 首页是一整张中国地图。
2. 点击地区进入图文旅行记录。
3. 计划页面使用日历。
4. 点击日期编辑计划。
5. 需要邀请另一半加入。
6. 点击地区后进入新页面，而不是弹窗或底部面板。
7. 地图体验参考百度地图的缩放和拖动手感。
8. 记录页面只有一个发布入口，图片和日记一起填写、一起发布。
9. 第一版一条记录最多 3 张图片。
10. 地图使用 Apache ECharts，可缩放、拖动。
11. 城市展示方式选择 A：点击省份下钻进入省内城市地图。
12. 缩放只用于查看细节，不自动切换省份/城市层级。

待确认：

1. 底部导航需要几个入口。
2. 是否需要“我们”的共同主页信息页。
3. 日历计划页是否按“旅行”为单位，还是全局日历。

## 4. 发布流程

已确认：

1. 第一版部署路线选择 B：前端单独部署，后端部署服务器，MySQL/Redis 用云数据库。
2. 部署方案应尽量贴近企业实践，兼顾就业展示价值。

待确认：

1. 前端部署平台选 Vercel、Cloudflare Pages、Netlify，还是云服务器 Nginx。
2. 后端部署平台选阿里云 ECS、腾讯云 CVM、轻量应用服务器，还是其他。
3. MySQL 使用云数据库 RDS，还是云服务器自建 MySQL。
4. Redis 使用云 Redis，还是云服务器自建 Redis。
5. 第一版是否先使用免费二级域名。
6. 是否购买独立域名。
7. 是否需要准备搜索引擎收录。
8. 是否需要中国大陆访问优化和备案规划。

## 5. 登录与双人互通

已确认：

1. 先不登录也能体验。
2. 后续需要邀请另一半加入。
3. 两个人可以共同上传照片、写日记、编辑旅行。
4. 第一版登录方式选择匿名体验 + 邀请码。
5. 邀请码有效期为 24 小时。
6. 一个情侣空间第一版限制 2 个人。
7. 匿名用户身份使用浏览器 localStorage 保存。
8. 第一版图片不做缩略图。
9. AI 润色日志保存完整输入和输出。
10. AI 日记润色限制为每人每天 10 次。
11. OSS 上传方式为前端拿签名后直传 OSS。
12. 删除旅行时，旅行、日记、图片元数据都做逻辑删除。
13. 删除图片时，同步删除 OSS 文件，避免后期存储堆积。

待确认：

1. 后续是否支持匿名用户绑定正式账号。
