# 多空间与空间切换实施计划

## 目标

把当前“单一情侣/个人空间”升级为第一版多空间能力：每个注册用户自动拥有个人旅行空间，同时最多拥有或加入一个 2 人情侣空间。用户在“我”页面顶部看到当前空间，点击进入空间切换页；切换后首页地图、城市日记、照片和计划页都使用当前空间的数据。情侣空间在伴侣加入前显示“未邀请你的伴侣不可使用。”，并禁止编辑记录与计划。

## 已确认产品规则

1. 用户注册成功后，系统自动创建个人空间。
2. 个人空间名称：`{用户名}的旅行空间`。
3. 情侣空间创建后，如果伴侣未加入，名称：`{用户名}的情侣旅行空间`。
4. 情侣空间伴侣加入后，名称自动变成：`{用户名1}和{用户名2}的旅行空间`。
5. 第一版情侣空间固定 2 人，不做 3-4 人家庭空间，不做无限空间。
6. 用户界面不主展示 `u_用户ID` 或 `r_随机编号`，只展示自然空间名、空间类型、状态、小图标。
7. 系统内部继续使用数字 `spaceId` 做稳定关联。
8. 个人空间与情侣空间数据完全隔离。
9. 切换空间后，地图、城市记录、照片、日记、计划全部切到当前空间。
10. 情侣空间未开启时可以创建、查看、生成邀请码、复制邀请码，但不能编辑地图记录、发布日记照片、编辑计划。
11. 计划页从日历改成 Day 列表，减少手机端占用。

## 当前代码现状

现有后端已经有这些基础：

- `trip.space_id`
- `trip_post.space_id`
- `trip_image.space_id`
- `travel_plan_day.space_id`
- `couple_space`
- `couple_space_member`
- `invite_code`

因此这次不是推倒重做，重点是补齐：

- 空间类型：个人 / 情侣
- 空间状态：已开启 / 等待伴侣加入
- 用户当前空间：`active_space_id`
- 注册后自动创建个人空间
- 创建情侣空间与切换当前空间接口
- 未开启情侣空间的编辑权限限制
- 前端空间切换页与当前空间展示
- 计划页 Day 列表改造

## 数据库改造

### 新增迁移脚本

创建：`data/sql/004_multi_space_current_space.sql`

建议内容：

```sql
USE love_travel;

ALTER TABLE app_user
  ADD COLUMN active_space_id BIGINT NULL AFTER status;

ALTER TABLE couple_space
  ADD COLUMN space_type VARCHAR(20) NOT NULL DEFAULT 'COUPLE' AFTER cover_image_url,
  ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE' AFTER space_type;

CREATE INDEX idx_active_space_id ON app_user (active_space_id);
CREATE INDEX idx_space_type_status ON couple_space (space_type, status);
```

如果本机 MySQL 不支持重复执行 `ADD COLUMN` 的幂等写法，就保持“一次性迁移脚本”，执行前先确认字段不存在。

### 新用户数据

注册时创建：

```text
couple_space.space_type = PERSONAL
couple_space.status = ACTIVE
couple_space.member_limit = 1
couple_space.space_name = {nickname}的旅行空间
couple_space_member.role = owner
app_user.active_space_id = personalSpace.id
```

### 旧用户兼容

登录或查询空间时，如果用户没有 `active_space_id`：

1. 查找其现有 membership。
2. 如果没有空间，创建个人空间。
3. 如果有一个单人成员旧空间，把它标记为 `PERSONAL / ACTIVE / member_limit=1` 并设为当前空间。
4. 如果有两人成员旧空间，把它标记为 `COUPLE / ACTIVE / member_limit=2` 并设为当前空间。

此兼容逻辑放在后端 `SpaceService.ensureActiveSpace(userId)` 中。

## 后端接口设计

### 1. 查询空间列表

```text
GET /api/spaces
```

返回：

```json
[
  {
    "spaceId": 1,
    "spaceName": "焦焦的旅行空间",
    "spaceType": "PERSONAL",
    "status": "ACTIVE",
    "creatorUserId": 1,
    "memberCount": 1,
    "memberLimit": 1,
    "current": true,
    "editable": true
  },
  {
    "spaceId": 2,
    "spaceName": "焦焦的情侣旅行空间",
    "spaceType": "COUPLE",
    "status": "WAITING",
    "creatorUserId": 1,
    "memberCount": 1,
    "memberLimit": 2,
    "current": false,
    "editable": false
  }
]
```

权限：只返回当前用户属于的空间。

### 2. 查询当前空间

```text
GET /api/spaces/current
```

返回同上单个对象。后端必须确保用户至少有个人空间。

### 3. 创建情侣空间

```text
POST /api/spaces/couple
```

请求体：

```json
{}
```

规则：

- 用户必须登录。
- 如果用户已经有一个未删除的情侣空间，直接返回该情侣空间，不重复创建。
- 创建后状态为 `WAITING`。
- 创建者加入成员表，role 为 `owner`。
- 不自动迁移个人空间数据。
- 可选择创建后自动切换当前空间到情侣空间，推荐自动切换，方便用户立刻复制邀请码。

返回：空间对象。

### 4. 切换当前空间

```text
POST /api/spaces/{spaceId}/activate
```

请求体：

```json
{}
```

规则：

- 用户必须是该空间成员。
- 更新 `app_user.active_space_id`。
- 返回切换后的空间对象。

### 5. 生成邀请码

```text
POST /api/spaces/{spaceId}/invite-code
```

规则：

- 只能对 `COUPLE` 空间生成。
- 当前用户必须是空间成员。
- 空间成员数必须小于 2。
- 邀请码 1 分钟过期。
- 邀请码不能重复。

### 6. 通过邀请码加入情侣空间

```text
POST /api/invite-codes/join
```

请求体：

```json
{
  "code": "A9Q2MA"
}
```

规则：

- 邀请码必须存在、未使用、未过期。
- 目标空间必须是 `COUPLE` 且成员数小于 2。
- 用户不能加入自己创建且已在其中的空间。
- 用户第一版最多只能有一个情侣空间。如果已经有其他情侣空间，返回错误。
- 加入成功后：
  - 新增 membership。
  - 邀请码标记 used。
  - 情侣空间状态改为 `ACTIVE`。
  - 空间名改成 `{创建者用户名}和{加入者用户名}的旅行空间`。
  - 加入者 `active_space_id` 切到该情侣空间。

## 后端服务改造

### SpaceService

新增核心方法：

```java
public CoupleSpaceResponse ensureActiveSpace(Long userId)
public List<CoupleSpaceResponse> listSpaces(Long userId)
public CoupleSpaceResponse createCoupleSpace(Long userId)
public CoupleSpaceResponse activateSpace(Long userId, Long spaceId)
public CoupleSpace requireEditableActiveSpace(Long userId)
```

`requireEditableActiveSpace` 规则：

- 个人空间可编辑。
- 情侣空间 `ACTIVE` 可编辑。
- 情侣空间 `WAITING` 不可编辑，抛出错误：`未邀请你的伴侣不可使用。`

### TravelService

改造点：

- `listVisitedRegions` 使用 `active_space_id`。
- `getTravelPage` 使用 `active_space_id`。
- `updateDateRange`、`createPost`、`updatePost`、`deletePost`、`deleteImage` 必须要求当前空间可编辑。
- 查询仍需校验 trip/post/image 属于当前空间。

### PlanService

改造点：

- `listDays` 使用当前空间。
- `createDay`、`updateDay`、`deleteDay` 使用 `requireEditableActiveSpace`。
- 删除日历相关后端字段不必做，保留 `plan_date`，前端改成 Day 列表即可。

### AuthService

注册成功后自动创建个人空间。

为了避免循环依赖，推荐抽出一个小服务：

```text
UserSpaceService
```

职责：

- `createPersonalSpaceForUser(AppUser user)`
- `ensureActiveSpaceForUser(Long userId)`
- `buildDefaultPersonalSpaceName(AppUser user)`

如果不想新增服务，也可以让 `AuthService` 依赖 `SpaceService`，但要注意不要让 `SpaceService` 反过来依赖 `AuthService`。

## 前端页面改造

### “我”页面 `/space`

顶部显示用户信息和当前空间卡片：

```text
当前空间
[个人图标] 焦焦的旅行空间
个人空间 · 已开启
点击切换
```

或者：

```text
当前空间
[情侣图标] 焦焦的情侣旅行空间
情侣空间 · 未邀请你的伴侣不可使用。
点击切换
```

点击当前空间卡片跳转 `/spaces`。

“我”页面保留设置入口，不再把所有空间创建/邀请码逻辑堆在第一页。

### 新增 `/spaces` 页面

页面标题：切换旅行空间。

展示：

- 个人空间卡片
- 情侣空间卡片，如果没有则显示“创建情侣空间”
- 当前使用中的空间显示“当前使用中”
- 非当前空间显示“切换到此空间”
- 等待伴侣加入的情侣空间显示“未邀请你的伴侣不可使用。”
- 情侣空间可生成邀请码、复制邀请码、显示倒计时
- 输入对方邀请码加入空间

### 计划页 `/plans`

去掉日历区域，改成 Day 列表：

```text
旅行计划
当前空间：焦焦的旅行空间
[新增一天]

Day 1
日期：2026-06-19
标题：到达青岛
安排：...
[编辑] [删除]
```

编辑方式：

- 点击 Day 卡片进入编辑状态。
- 按“确定保存”才调用接口。
- 按“取消”恢复原数据。
- 删除前必须确认。

如果当前空间不可编辑：

- 新增、保存、删除按钮禁用。
- 页面顶部显示“未邀请你的伴侣不可使用。”

### 城市记录页

如果当前空间不可编辑：

- 日期保存按钮禁用。
- 发布记录按钮禁用。
- 图片上传禁用。
- 编辑/删除按钮禁用。
- 页面顶部显示“未邀请你的伴侣不可使用。”

仍允许查看空状态和空间信息。

## 实施顺序

1. 数据库迁移和实体字段。
2. 后端空间响应对象与 SpaceService 多空间能力。
3. 注册自动创建个人空间和旧用户兼容。
4. 后端旅行记录、地图、计划统一使用当前空间，并限制 WAITING 情侣空间编辑。
5. 前端封装 spaces API。
6. “我”页面当前空间入口。
7. 新增 `/spaces` 切换空间页。
8. 计划页改 Day 列表。
9. 城市记录页补不可编辑状态。
10. 跑后端测试、前端构建，最后手动联调。

## 风险和注意事项

1. 当前仓库中文内容在终端中可能显示乱码，但不一定代表文件坏了；修改文件优先用 `apply_patch`。
2. MySQL 迁移脚本不要使用当前环境不支持的 `ADD COLUMN IF NOT EXISTS`。
3. 旧数据不要删除，个人空间和情侣空间数据应隔离，不做迁移合并。
4. 后端不能相信前端传入的 `spaceId` 来判断权限；所有核心查询都以登录用户的 `active_space_id` 和 membership 校验为准。
5. 邀请码仍然只用于加入空间，不等于空间 ID。
6. 等待状态情侣空间的提示文案统一为：`未邀请你的伴侣不可使用。`

## 完成标准

- 新用户注册后自动拥有个人空间。
- 用户可以创建并切换情侣空间。
- 未开启情侣空间不可编辑记录和计划。
- 伴侣加入后情侣空间可编辑。
- 空间切换后地图、城市记录、计划数据完全隔离。
- 计划页不再使用日历，改成 Day 列表。
- 前端构建通过。
- 后端测试或 Maven 打包通过。
