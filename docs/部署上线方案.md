# 部署上线方案

更新时间：2026-06-24

## 1. 当前建议

当前不建议直接做正式公开上线，建议先做一次“测试部署”。

测试部署目标：

1. 让项目离开本地环境，在服务器上跑起来。
2. 学会前端、Java 后端、Python AI 服务、MySQL、Redis、OSS、Nginx 的协作方式。
3. 提前发现跨域、环境变量、端口、HTTPS、文件上传大小、服务重启等线上问题。
4. 形成可以写进简历和面试的“项目部署经验”。

正式上线目标：

1. 给真实用户长期使用。
2. 考虑域名、HTTPS、备案、数据备份、监控、隐私协议、成本控制。

## 2. 当前项目服务拆分

| 模块 | 当前技术 | 线上职责 |
| --- | --- | --- |
| 前端 | Next.js | 用户页面、地图、记录、计划、设置、AI 弹窗 |
| Java 后端 | Spring Boot | 登录、空间、权限、旅行记录、计划、OSS、AI 统一入口 |
| Python AI 服务 | FastAPI / LangChain | 调用通义千问，生成 AI 旅行规划 |
| MySQL | MySQL | 保存用户、空间、记录、计划、AI 运行记录 |
| Redis | Redis | Session、邀请码、登录失败限制、AI 防重复与限流 |
| OSS | 阿里云 OSS | 保存旅行图片 |
| Nginx | Nginx | 反向代理、静态资源、HTTPS、统一入口 |

## 3. 测试部署推荐方案

### 3.1 免费优先原则

当前阶段优先使用免费资源或免费额度。

详细筛选见：

```text
docs/免费部署资源筛选清单.md
```

原则：

1. 能免费先免费。
2. 免费方案只用于测试部署和学习，不承诺长期稳定。
3. 不把正式用户数据放在不确定的免费环境里。
4. 免费额度、休眠规则、构建次数、流量限制会变化，真正使用前必须看平台官网最新说明。
5. API Key、数据库密码、OSS 密钥仍然不能因为是免费环境就随便写进代码。

可优先考虑的免费或低成本方向：

| 模块 | 免费优先思路 | 注意事项 |
| --- | --- | --- |
| 前端 | Cloudflare Pages / Vercel / Netlify 免费额度 | 免费额度和构建限制以官网为准 |
| 后端 | 本地穿透测试或云厂商免费试用服务器 | 免费试用可能有期限 |
| MySQL | 测试阶段先用服务器自建 MySQL | 免费但需要自己备份和维护 |
| Redis | 测试阶段先用服务器自建 Redis | 必须禁止公网访问 |
| OSS | 使用已有阿里云 OSS，小流量测试 | 存储和流量可能产生少量费用 |
| AI | 使用阿里云百炼免费额度或控制调用次数 | 免费额度可能过期，必须限流 |
| 域名 | 先用平台二级域名或服务器 IP | 正式域名通常按年收费 |
| HTTPS | 优先使用平台自动 HTTPS 或免费证书 | 自建 Nginx 可后续配置免费证书 |

### 3.2 推荐方案 A：低成本学习版

用一台云服务器跑大部分服务：

```text
用户浏览器
  -> Nginx
    -> Next.js 前端
    -> Java Spring Boot
    -> Python FastAPI
  -> MySQL
  -> Redis
  -> 阿里云 OSS
```

其中 MySQL 和 Redis 可以先装在同一台服务器上。

优点：

1. 成本低。
2. 部署链路简单。
3. 很适合第一次学习上线。
4. 能完整理解 Linux、Nginx、端口、进程、环境变量。

缺点：

1. 不适合大量用户。
2. MySQL、Redis 和业务服务在同一台机器，稳定性不如企业正式架构。
3. 需要自己做备份和安全配置。

适合当前阶段：是。

### 3.3 方案 B：更接近企业版

```text
用户浏览器
  -> 前端平台或 Nginx
  -> Java 后端服务器
  -> Python AI 服务
  -> 云数据库 RDS
  -> 云 Redis
  -> 阿里云 OSS
```

优点：

1. 更接近企业真实生产架构。
2. MySQL、Redis 稳定性更好。
3. 后续扩容更容易。

缺点：

1. 成本更高。
2. 第一次部署配置项更多。
3. 对新手不如方案 A 直观。

适合当前阶段：后续正式上线或面试展示增强阶段。

## 4. 当前建议执行路线

建议先走：

```text
本地 V1 完成
  -> 免费优先筛选
  -> 测试部署方案 A
  -> 写部署复盘和面试题
  -> 再考虑方案 B 或正式上线
```

理由：

1. 你还没完整部署过，先用低成本方案把流程跑通更重要。
2. 当前项目还在持续扩展，不适合马上花较多成本上正式生产配置。
3. 测试部署成功后，再迁移到 RDS、云 Redis、HTTPS、正式域名会更稳。
4. 如果有免费平台或免费试用，就先用免费资源验证链路。

## 5. 测试部署服务器目录建议

建议服务器上使用统一目录：

```text
/opt/love-travel/
  front/
  server/
  ai-service/
  logs/
  env/
  scripts/
  backup/
```

目录用途：

| 目录 | 用途 |
| --- | --- |
| `/opt/love-travel/front` | 前端构建产物或 Next.js 服务 |
| `/opt/love-travel/server` | Java 后端 jar、配置、启动脚本 |
| `/opt/love-travel/ai-service` | Python AI 服务代码、虚拟环境、启动脚本 |
| `/opt/love-travel/logs` | 前端、Java、Python、Nginx 相关日志 |
| `/opt/love-travel/env` | 环境变量文件，禁止提交到 GitHub |
| `/opt/love-travel/scripts` | 启动、停止、重启、备份脚本 |
| `/opt/love-travel/backup` | MySQL 备份、部署前备份 |

## 6. Nginx 路由设计

测试环境可以先这样设计：

```text
https://你的域名/
  -> 前端

https://你的域名/api/
  -> Java Spring Boot 8080

https://你的域名/internal-ai/
  -> Python FastAPI 8000
```

注意：

1. Python AI 服务不建议直接暴露给普通用户。
2. 前端只应该请求 Java 后端。
3. Java 后端再调用 Python AI 服务。
4. 线上可以让 Python 只监听 `127.0.0.1:8000`，不对公网开放。

### 7.2 Python AI 服务

```text

```

### 7.3 前端

```text
NEXT_PUBLIC_API_BASE_URL
```

如果前端和后端通过同一域名由 Nginx 转发，可以让前端请求 `/api`，减少跨域问题。

## 8. 数据库与 Redis 部署建议

### 8.1 测试部署

测试部署可以先在同一台服务器自建：

1. MySQL。
2. Redis。

上线前必须做：

1. MySQL 设置强密码。
2. Redis 不直接暴露公网。
3. 只开放必要端口。
4. 定期备份 MySQL。

### 8.2 正式上线

正式上线建议升级为：

1. 云数据库 RDS。
2. 云 Redis。
3. 自动备份。
4. 访问白名单。

## 9. OSS 部署建议

当前项目已经使用阿里云 OSS。

建议：

1. 继续使用独立 Bucket。
2. 不把 AccessKey 写进代码。
3. 通过环境变量配置 AccessKey。
4. 后续如果流量变大，可以考虑 CDN。
5. 删除图片时同步删除 OSS 对象，避免存储费用堆积。

## 10. 安全检查清单

测试部署前必须检查：

```text
[ ] GitHub 或公开仓库中没有 AccessKey、数据库密码、DashScope API Key
[ ] Java 后端使用环境变量读取密钥
[ ] Python AI 服务使用环境变量读取 API Key
[ ] Redis 不暴露公网
[ ] MySQL 不暴露公网，或仅允许可信 IP 访问
[ ] Nginx 只开放 80/443
[ ] Java 后端不直接暴露不必要端口
[ ] Python AI 服务不直接暴露公网
[ ] OSS Bucket 权限符合当前上传访问策略
[ ] 上传文件大小有限制
[ ] 登录失败限制已启用
[ ] AI 接口限流已启用
```

## 13. 成本理解

部署不是按“部署一次”收费。

主要成本来自：

1. 云服务器运行时长。
2. MySQL 或 RDS 运行时长。
3. Redis 或云 Redis 运行时长。
4. OSS 存储容量和访问流量。
5. AI 接口调用量。
6. 域名年费。
7. 可能的 HTTPS、CDN 或备案相关成本。

测试部署阶段尽量减少成本：

1. 优先找免费额度或免费试用。
2. 先用一台低配服务器或免费试用服务器。
3. MySQL 和 Redis 先自建。
4. 前端优先使用免费前端托管平台，或者先跑在同一台服务器。
5. AI 功能保留限流和额度。
6. 不急着购买高配云数据库。
7. 不急着购买正式域名，可以先用平台二级域名或服务器 IP 测试。
