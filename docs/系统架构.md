# 系统架构

## 1. 已确认架构

本项目采用“前端 + Java 业务后端 + Python AI 服务”的分层架构。

```text
前端 Next.js
  -> Java Spring Boot 统一业务后端
  -> Python FastAPI AI 服务
  -> LangChain / 大模型 API

Java Spring Boot
  -> MySQL
  -> Redis
  -> 阿里云 OSS
```

## 2. 为什么前端只调用 Java

已确认采用方案 B：

```text
前端只调用 Java，Java 再调用 Python AI 服务。
```

原因：

1. 前端只需要维护一个主要后端入口。
2. 用户登录、权限、情侣空间校验统一在 Java 后端完成。
3. AI 调用次数限制可以放在 Java 后端和 Redis 中处理。
4. Python AI 服务可以专注 AI 能力，不直接处理复杂业务权限。
5. 更贴近企业中的多服务协作模式。

## 3. 服务职责

| 服务 | 技术 | 职责 |
| --- | --- | --- |
| 前端 | Next.js | 页面、交互、地图、日历、上传入口 |
| 业务后端 | Spring Boot | 用户、情侣空间、旅行、记录、计划、权限、OSS、AI 统一入口 |
| AI 服务 | FastAPI | 日记润色、旅行总结、计划建议、后续 LangChain 编排 |
| 数据库 | MySQL | 保存核心业务数据 |
| 缓存 | Redis | 验证码、邀请状态、限流、短期缓存 |
| 对象存储 | 阿里云 OSS | 保存旅行照片 |

## 4. AI 调用流程示例

用户点击“AI 润色”：

```text
1. 前端请求 Java：POST /api/ai/polish-diary
2. Java 校验用户身份和情侣空间权限。
3. Java 检查 Redis 中的 AI 调用次数限制。
4. Java 调用 Python FastAPI：POST /internal/ai/polish-diary
5. Python 调用 LangChain / 大模型 API。
6. Python 返回润色结果给 Java。
7. Java 保存必要记录并返回给前端。
8. 前端展示润色后的文字。
```

## 5. 第一版最小 AI 入口

第一版只预留一个最小 AI 能力：

```text
AI 日记润色
```

第一版已确认直接接真实大模型 API，不使用模拟结果。

第一版需要做成：

1. 前端有“AI 润色”按钮。
2. Java 有 AI 润色接口。
3. Python FastAPI 有 AI 润色接口。
4. Python FastAPI 通过 LangChain 或直接 SDK 调用真实大模型 API。
5. Java 后端需要处理 AI 调用权限、调用次数限制、失败兜底和日志记录。
6. API Key 必须放在后端环境变量中，不能写进前端代码。
